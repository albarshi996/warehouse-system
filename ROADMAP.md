## 5. المتبقّي في الواجهة والتكامل (الربط بأودو + نظام الوصول)

### 5.1 التكامل مع أودو الحقيقي (S8) — الطبقة جاهزة، ينقصها التفعيل
تفاصيل كاملة في [`ODOO_INTEGRATION_HANDOFF.md`](ODOO_INTEGRATION_HANDOFF.md). باختصار المتبقّي:
1. نشر الوسيط (Cloudflare Worker أو Firebase) بأسراره الأربعة (ODOO_URL/DB/USER/API_KEY).
2. توسيع `ALLOWLIST` للنماذج/الدوال التي تحتاجها الشاشات.
3. `PUBLIC_ODOO_MODE=production` + `PUBLIC_ODOO_PROXY_URL` ثم إعادة بناء ونشر.
4. تحويل الشاشات إلى أودو تدريجيًّا (المنتجات ثم الحركات).

### 5.2 نظام الوصول (من حاوية `/dashboard/access-control`) — النقاط القادمة
1. **تسجيل الدخول الفعلي** (Firebase Auth) — شاشة دخول عربية للبوابة. *(النقطة الأولى المقترحة للبدء)*
2. **جلب الأدوار من أودو** — قراءة مجموعات المستخدم عبر الوسيط وتخزين الدور في الجلسة.
3. **قائمة جانبية حسب الدور** — إخفاء/إظهار مجموعات القائمة تلقائيًّا حسب الصلاحية.
4. **صفحة إدارة المستخدمين** — إسناد الأدوار وتعطيل الحسابات من داخل البوابة.
5. **سجلّ التدقيق** — من فعل ماذا ومتى.

## 8. الخطة الاستراتيجية الأمامية — الركائز الثماني (خطة 2026) 🚀

> **ميثاق الشراكة (2026-07-13):** المالك يحلم ويختار ويتعلّم؛ القائد البرمجي يخطّط وينفّذ ويُلهم.
> **الهدف الأسمى:** نظام مستودعات **مثالي واحترافي — الأول من نوعه في ليبيا**، جاهز ليكون منتجًا.
> كل ركيزة تُعرَض بـ: **الهدف · ما لدينا · الخطوات · توصيتي · قرارك المطلوب.**

### ⬛ ركيزة 1 — نظام الأدوار والصلاحيات (من يرى ماذا · من يفعل ماذا)

**الهدف:** لكل موظف واجهة تُظهر ما يخصّه فقط، وصلاحيات تسمح بأفعاله فقط — بحيث تُفرَض الحرّاس الستة على مستوى الدور لا بالنيّة.
**ما لدينا:** 7 مجموعات أمان في الموديول + حاوية `/dashboard/access-control`. **✅ منجز ومنشور حيًّا (2026-07-13):** (أ) **M1:** Firebase Auth + شاشة دخول `/login` + حارس `AuthGate` + 10 أدوار + بطاقة المستخدم/الخروج + تمهيد أدمن بالبريد (المالك دخل كـ«👑 المدير العام»). (ب) **الإكمال (`d06bb98`):** تقييد القائمة حسب الدور (`navAccess.js` + `RoleNav`) + صفحة إدارة المستخدمين `/dashboard/users` (إسناد الأدوار الـ10 + تفعيل/تعطيل + إضافة بالـUID، أدمن فقط). **✅ قواعد أمان Firestore نُشرت (2026-07-15):** [`firestore.rules`](firestore.rules) — كانت القاعدة **مفتوحة للعالم** (ثبت بالاختبار: قراءة/كتابة/حذف بلا حساب)؛ الآن: لا وصول بلا دخول · الأدوار للأدمن فقط · سجلّ المسح غير قابل للتعديل/الحذف · منع افتراضي. أُعيد الاختبار = كل المحاولات مرفوضة. **يتبقّى:** جلب الأدوار من أودو + شاشة متابعة العمليات للمدير.

**مصفوفة الأدوار (المرجع المعتمد):**

| الدور | يرى (شاشات) | يفعل (أفعال محكومة) | المجموعة |

| **مدير المستودع** | كل شاشات المستودع + التقارير + لوحات KPI | يعتمد، يوزّع المهام، يراقب | `warehouse_user`+ |

| **مفتّش الجودة** | الاستلامات المعلّقة · تقرير الجودة | يوافق/يرفض الجودة 🥇 (يفكّ حجب Done) | `qc_inspector` |

- **«يرى ماذا» ← الواجهة:** تُخفى/تُظهر مجموعات القائمة الجانبية (`navGroups`) تلقائيًّا حسب الدور.

- **منطق الأدمن 7777 (قرار المالك المؤجّل):** رمز دخول موحّد يمنح رؤية شاملة — يُدمَج كطبقة أدمن فوق المصفوفة.

**توصيتي:** نبني هذا فوق **Firebase Auth** (الدخول) + جدول أدوار، ثم لاحقًا نجلب الدور الحقيقي من مجموعات أودو عبر الوسيط.
**قرارك:** هل نبدأ بمنطق الأدمن 7777 المبسّط أولًا (سريع)، أم بالمصفوفة الكاملة مباشرة؟

مفصّل في §4. باختصار: **(أ) S12 الإغلاق المالي** (المرحلة 12/12 الأخيرة — `bz.period.close` + أقفال الفترات + قائمة إغلاق) · **(ب) تشغيل الاختبارات الـ42 فعليًّا** في بيئة أودو حقيقية (تتحقّق بركيزة 3) · **(ج) تعميقات اختيارية:** ASN صريح، Bin Card، سجلّ تدقيق كامل · **(د) S8** ربط الموديول بالواجهة.
**توصيتي:** ننجز S12 **بعد** توفّر بيئة أودو حيّة (ركيزة 3) لنختبره فعليًّا لا ساكنًا فقط.

### ⬛ ركيزة 3 — قاعدة بيانات حيّة (أفضل الخيارات المجانية) 💾

**الهدف:** بيئتان حيّتان مجانيتان: (أ) قاعدة الواجهة/الدخول، (ب) أودو حقيقي يشغّل موديولنا المخصّص.
**الحقيقة الحاكمة:** موديول `brandzo_warehouse` **مخصّص** — لا يعمل على «Odoo Online» المجاني (لا يقبل موديولات مخصّصة). فنحتاج استضافة ذاتية.

| الخيار | يصلح لـ | المجانية | الحكم |
|---|---|---|---|
| **Oracle Cloud «Always Free»** (ARM حتى 4 أنوية/24GB) | أودو 19 + PostgreSQL (استضافة ذاتية Docker) | مجاني **للأبد** | ★ **توصيتي** — يشغّل موديولنا المخصّص بقوة إنتاجية |
| **Firebase (خطة Spark)** | دخول Auth + بيانات الواجهة | مجاني سخيّ | ★ **مدمج أصلًا** — نبدأ به فورًا |
| Supabase / Neon (Postgres) | بديل لبيانات الواجهة | 500MB مجاني | بديل جيد لو أردنا علائقيًّا |
| Odoo Online المجاني | تطبيق واحد فقط | مجاني | ✗ **لا يقبل موديولنا** |
| Google/AWS free | أودو | صغير/12 شهرًا | ✗ ضعيف أو مؤقّت |

**توصيتي (المعمارية المجانية الكاملة):**
`الواجهة + الدخول → Firebase` · `أودو الحقيقي + موديولنا → Oracle Always Free (Docker: Odoo19 + Postgres)` · `الجسر بينهما → odoo-proxy`.
**قرارك:** (أ) نبدأ بـ **Firebase فقط** كـ MVP سريع، أم (ب) نؤسّس **Oracle + أودو حيّ** من البداية للقوة الكاملة؟

## 11. خطة النماذج الـ21 — من ورقٍ مطبوع إلى نظام مستندي حيّ 📑

> **الحالة (فحص فعلي 2026-07-15):** `public/forms/` فيه **22 نموذجًا** (1.2MB) + ملفان مشتركان
> (`brandzo-forms.css` · `brandzo-letterhead.js` — تستخدمهما **كل** النماذج ⇒ مَشْبَك مركزي جاهز).
> نموذج الاستلام وحده **1887 سطرًا و119 حقل إدخال**.

### 11.1 التشخيص — لماذا هي «ورق داخل متصفّح»
| العيب | الأثر التشغيلي |
|---|---|
| **صفر حفظ** (لا localStorage ولا قاعدة بيانات) | تملأ النموذج ثم تُغلق الصفحة ⇒ **ضاع كل شيء**. الوظيفة الوحيدة `window.print()`. |
| **أرقام مستندات مكتوبة في الكود** (`GRN-002` ثابت) | كل استلام مطبوع يحمل **نفس الرقم** ⇒ لا تسلسل ولا تتبّع ولا تدقيق. |
| **لا هوية ولا صلاحية** | لا يُعرف من ملأ المستند ولا من اعتمده — والاعتماد توقيع بالقلم. |
| **لا ترابط بين المستندات** | أمر الشراء والاستلام والفاتورة أوراق منفصلة ⇒ **المطابقة الثلاثية يدوية**. |
| **معزولة عمّا بنيناه** | لا تعرف الدخول ولا الأدوار ولا الجرد السحابي ولا الحرّاس الستة. |

### 11.2 المبدأ الحاكم: **محرّك واحد لا 21 تطبيقًا**
النماذج ليست 21 شاشة مستقلّة — بل **طبقة إدخال للدورة المستندية الـ12** التي يفرضها موديول أودو ويعلّمها المحاكي.
لذا نبني **محرّك مستندات واحدًا** + «مخطّط» (schema) لكل نموذج (حقوله · أعمدة جدوله · قواعده · دوره)،
ويتكفّل المحرّك بـ: مسودّة/حفظ · **ترقيم تسلسلي حقيقي** · هوية ووقت · اعتماد حسب الدور · **سجلّ تدقيق دائم** · طباعة.
**التصميم المطبوع الجميل يبقى كما هو** — لكن يُطبع من بيانات محفوظة برقم حقيقي وتوقيع معروف.

### 11.3 نموذج البيانات المقترح (يمتدّ من `operations` الذي نجح)
```
documents/{docId}
   type    : 'GRN' | 'PO' | 'GatePass' | …        ← أيّ النماذج الـ21
   number  : 'GRN-2026-0001'                       ← تسلسل حقيقي (لا يتكرّر)
   stage   : 04                                    ← ربطه بالدورة الـ12
   state   : draft → submitted → approved → done
   header  : {…}    lines: [{…}]                   ← حقول الرأس وبنود الجدول
   links   : { poId, grnId, … }                    ← سلسلة المستندات (أساس المطابقة الثلاثية)
   createdByUid/Name · createdAt · updatedAt
   └── audit/{id}                                  ← append-only: من فعل ماذا ومتى (نفس نمط scans)
```

### 11.4 الربط بما بنيناه فعلًا (القيمة الكبرى)
1. **الهوية والأدوار** (جاهزة): مفتّش الجودة يعتمد تقرير الجودة · ضابط البوابة يُصدر التصريح · المدير المالي يعتمد التسوية.
2. **🔥 الجرد الذكي ← نموذج الاستلام:** تمسح الأصناف بالباركود فتملأ **بنود GRN تلقائيًّا** بدل 119 حقلًا يدويًّا. (أقوى تكامل ممكن — والبنية جاهزة.)
3. **الحرّاس الستة** تصير حقيقية في الواجهة: لا استلام «تمّ» قبل اعتماد الجودة · لا خروج بلا تصريح معتمد · لا ترحيل قبل المطابقة الثلاثية.
4. **`odooMapper.js`** (موجود): أسماء الحقول القياسية ⇒ المستندات تُرحّل لأودو لاحقًا بلا إعادة بناء.
5. **قواعد أمان Firestore**: نفس نمط `scans` — إضافة فقط، لا محو للتاريخ.
